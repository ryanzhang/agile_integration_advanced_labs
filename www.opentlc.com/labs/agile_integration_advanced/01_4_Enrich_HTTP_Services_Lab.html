Enrich Content and Aggregate Data Lab

In this lab, you develop routes that process the account records in JSON format, and through enrichment and aggregation create a more complete account record. The resulting record is stored in a database.

Goals

  • Design Apache Camel routes that consume JSON records from AMQ

  • Use the multicast EIP pattern to distribute the messages to the services

  • Design the CXF web service using JAX-WS annotations to consume from SOAP backend service

  • Consume from the backend REST service

  • Aggregate the messages received using the aggregation EIP pattern

  • Save the aggregated message in an SQL database using the Spring SQL component

1. Import the Project into JBoss Developer Studio

  1. Using the same method you used in the previous lab, import the 04_enrich-content-REST-and-WS-microservices use case into a new workspace. You will find 3 projects:

    • soap-cxfws-service

    • rest-cxfrs-service

    • amq-enrich-persist-lab

2. Develop the Solution

2.1. REST Enrich Microservice

A REST microservice is used as a backend service for this solution. The REST service is provided in the project rest-cxfrs-service.

2.1.1. Inspect the REST Service

The REST service enriches the Account record.

  1. It consists of the CustomerRest REST service with the following annotations:

    @Path("/customerservice/")
public interface CustomerRest {

@POST @Path("/enrich") @Consumes("application/json")
Account enrich(Account customer);

    • The implementation class contains logic to enrich the Account POJO as follows:

      • Based on the geo field of the company object, a Java enumerator maps region codes to regions

      • {"NE,MW,SO,WE"} corresponds to {"NORTH_EAST,MIDWEST,SOUTH,WEST"}

      • The enriched Account object is returned by the enrich() method

    • To expose the REST service within Spring, you use the CXF component’s rsServer server. This is defined in the Application.java.

    • Check the application.properties and notice the cxf-path and server.port.

2.1.2. Run the REST service

  1. To run the REST service locally, run the following maven command from the project root folder:

    mvn spring-boot:run -Dfabric8.skip

  2. The REST service should be running on port 8080, and could be accessed through URL: http://localhost:8080/rest/customerservice/enrich

  3. Try a sample request to the REST endpoint:

    curl -k http://localhost:8080/rest/customerservice/enrich -X POST  -d '{"company":{"name":"Rotobots","geo":"NA","active":true},"contact":{"firstName":"Bill","lastName":"Smith","streetAddr":"100 N Park Ave.","city":"Phoenix","state":"AZ","zip":"85017","phone":"602-555-1100"}}' -H 'content-type: application/json'

  4. The response should be as follows:

    {"clientId":0,"salesRepresentative":null,"company":{"name":"Rotobots","geo":"NORTH_AMERICA","active":true},"contact":{"firstName":"Bill","lastName":"Smith","streetAddr":"100 N Park Ave.","city":"Phoenix","state":"AZ","zip":"85017","phone":"602-555-1100"}}
    Notice that the content is enriched, and the GEO is replaced by the correct location.

2.1.3. Deploy to OpenShift

  1. To deploy the application to OpenShift, execute the following maven command from the terminal:

    mvn fabric8:deploy

  2. Check that the project is deployed successfully. A pod for the deployment rest-cxfrs-service should be started. Notice the URL of the endpoint for external traffic.

  3. You can test the REST service by running the same curl request above, replacing the localhost URL with the remote URL. The response received should be the same as in localhost.

2.2. SOAP Web Service

A SOAP Web Service is provided as part of the assets for this lab. This service is developed using CXF-RS, and deployed on Karaf. The service is running on Fuse 6.3. This service is available in the project soap-cxfws-service.

2.2.1. Inspect the Solution

  1. A service is defined in the CustomerWS interface using the @Webservice annotation that exposes an operation to update the received Account object:

    @WebService
public interface CustomerWS {

    CorporateAccount updateAccount(Account account);

  2. Based on the Account POJO received, it creates a CorporateAccount POJO containing the same fields as Account plus these additional fields:

    • id: (int) Set as a random value using 1 to 100 as the range

    • salesContact: (String) Randomly populated using a list of contacts

  3. Uses the updateAccount method implemented within the CustomerWSImpl class to return CorporateAccount.

  4. Uses an Apache Camel CXF endpoint bean to configure the service and create it using these parameters:

    • id: customerWebService

    • address: http://localhost:9090/ws/customerService

    • serviceClass: org.fuse.usecase.service.CustomerWS

    • loggingFeatureEnabled: true

2.2.2. Run the SOAP service

  1. To run the SOAP service locally, run the following maven command from the project root folder:

    mvn camel:run -Dfabric8.skip

  2. The Camel service should be running on port 9090, and could be accessed through URL: http://localhost:9090/ws/customerService

  3. You can try a sample request using SOAP UI.

    <soapenv:Envelope xmlns:soapenv="http://schemas.xmlsoap.org/soap/envelope/" xmlns:ser="http://service.usecase.fuse.org/">
   <soapenv:Header/>
   <soapenv:Body>

               <active>true</active>

               <geo>NA</geo>
               <!--Optional:-->

  4. The response should be as follows:

2.3. Develop the Integration Module

Now you will be implementing the camel routes for enriching the Account data with the 2 bakend services, aggregate the responses, and finally persist the data into a database table. You will be completing the code for this section in the project amq-enrich-persist-lab.

2.3.1. H2 database

As part of this integration, you will be writing the records into a database. For the purposes of this lab, an embedded H2 database is provided to run the solution on localhost. You can find the configuration details in the pom.xml & the application-dev.properties.

An SQL script for creating the USECASE schema and the table required is provided in the schema.sql file.

You can access the H2 console by opening the URL : http://localhost:8080/h2-console . Please enter the JDBC url as jdbc:h2:mem:testdb and login.

The console will be available only after the routes are started. As the database is embedded, it does not persist data once the route is shutdown.

2.3.2. Develop the Camel routes

  1. Based on the code provided, create a new route that consumes the JSON objects created in the previous exercise. You can have your route start from the accountQueue.

  2. Set up the AMQP configuration & queues based on the previous lab.

  3. Convert the content to a string and unmarshall the JSON content to a POJO using the <unmarshal> processor.

    • JSON library: jackson

    • unmarshalTypeName attribute: org.globex.Account

    • The result is used to multicast the exchange to the REST and web service endpoints

      Do not multicast the exchange in parallel. To simplify the readability of the code, use a direct endpoint (callRestEndpoint) to call the REST service and a different direct endpoint (callWSEndpoint) to call the web service service.

2.3.3. Define the REST Direct Routes

  1. To call the backend services, create two routes: one starting with direct:callRestEndpoint and the other with direct:callWSEndpoint.

    • In order to correctly invoke the declared REST service, use a http producer component and make sure the following {header,value} pairs are specified:

      • Content-Type=application/json

      • Accept=application/json

      • CamelHttpMethod=POST

      • CamelCxfRsUsingHttpAPI=True

2.3.4. Define the Web Service Direct Route

  1. Develop a direct web service route that starts with the direct://callWSEndpoint endpoint and calls the web service using the Camel CXF Producer.

2.3.5. Design the Aggregation Strategy

  1. From the results collected by the two services, aggregate the content in the multicast processor in order to reconcile the information received from the CorporateAccount POJO with the Account POJO.

    • This new Account POJO contains the aggregated result that is sent to another Apache Camel route to insert the row into the database.

      The aggregation strategy must be developed within the AccountAggregator class.

2.3.6. Insert the Aggregated Data into the H2 Database

Next you create a new Apache Camel route to insert the Account information into the T_ACCOUNT table of the ACCOUNT database.

  1. Use the Spring SQL component and specify the SQL query to be used to insert the Account record within the option parameter of the SQL endpoint.

    • Method: defineNamedParameters(Account account)

  2. Define the named parameters using a Map<String, Object> that you create within a bean that is called by the route to set the body before calling the SQL endpoint.

    • Bean: ProcessorBean.class

  3. After calling the SQL endpoint, add a log message processor to display the contents of the received body.

2.3.7. Develop a JUnit Test

  1. Develop a JUnit test case to validate that the jackson ObjectMapper can read the string content of the JSON Account message and generate an Account object.

    • JUnit test class: JacksonCompanyTest

  2. Define two assert conditions: one to check that the object is not null and the other to verify that the name of the company is equal to "Robocops".

2.3.8. Build the routing Project and Test it Locally

  1. On the command line, run the following command in a terminal:

    mvn spring-boot:run -Dfabric8.skip

    • If your code is correct, you see output similar to the following:

      {CONTACT_STATE=NC, SALES_CONTACT=Guillaume Nodet, CREATION_USER=fuse_usecase, CREATION_DATE=2015-12-11 18:18:01.55, CONTACT_PHONE=600-555-7000, CONTACT_CITY=Raleigh, CONTACT_ADDRESS=1101 Smith St., COMPANY_NAME=MountainBikers, COMPANY_ACTIVE=true, CLIENT_ID=46, CONTACT_LAST_NAME=Jungle, CONTACT_ZIP=27519, COMPANY_GEO=SOUTH_AMERICA, CONTACT_FIRST_NAME=George}

  2. Check the database to ensure the records are populated correctly in the T_ACCOUNT table.
