Publish Records and Fix Errors Lab

In this lab you modify the use case you developed in the Split and Transform Rest lab to publish the individual JSON Account records to message input queues, including an error queue in the event of a processing error. You also develop a strategy to correct erroneous messages and redeliver them.

The route implements an interception strategy when IllegalArgumentException is thrown. The erroneous record is published to a special topic and an error code and error message are added.

A separate route listens for error messages. When this route is triggered, the erroneous message and message code are saved into an error table with the status of ERROR. To correct the record, you run a script against the error table and modify the status to FIXED. You use another route to poll the database for FIXED records. This route republishes the message on the original queue and updates the status to CLOSED.

Goals

  • Publish records from the processing to an error topic

  • Implement a custom error handling strategy to capture the erroneous messages

  • Fix the messages and republish them

1. Import the Project into JBoss Developer Studio

  1. Using the same method you used in the previous labs, import the 03_rest-publish-and-fix-errors use case into a new workspace.

2. Develop the Solution

2.1. H2 database

As part of this integration, you will be writing the records into a database. For the purposes of this lab, an embedded H2 database is provided to run the solution on localhost. You can find the configuration details in the pom.xml & the application-dev.properties.

An SQL script for creating the USECASE schema and the tables required is provided in the schema.sql file.

You can access the H2 console by opening the URL : http://localhost:8080/h2-console . Please enter the JDBC url as jdbc:h2:mem:testdb and login.

The console will be available only after the routes are started. As the database is embedded, it does not persist data once the route is shutdown.

2.2. Review the Existing Camel Routes

Because you are starting this project from Use Case 1, you need to use the code created for this project to replace the file component with the ActiveMQ component and the queues.

It is a good idea to create a Camel Spring JUnit test case to validate the logic of your Camel routes.

2.3. Implement the Interception Strategy

  1. Develop your interception strategy to fetch the IllegalArgumentException exception and move the information into a errorNotification topic with the associated error-code and error-message.

    • You can set the error code to 111 so that the error message corresponds to the first message reported by the stack trace that Camel Bindy throws.

    • The string content of the message must be saved into the JMS message so it can be used to update the T_ERROR table.

  • You can create a direct:error route that you call from your interception strategy.

  • It is a good idea to add an EIP log processor to collect information using the DEBUG logging level and the org.fuse.usecase log name.

2.4. Extend the Existing Camel Routes

  1. Add a new route with an ActiveMQ endpoint to consume the JMS messages from the errorNotification topic.

  2. Develop your strategy to insert a record with the information from this route (code, error message, message, and status) into the T_ERROR table.

2.5. Redeliver the Corrected Message

  1. Based on the information saved into the T_ERROR database, develop your strategy to inject the corrected message into the inputQueue queue.

Remember to update the status according to the values defined (a new message has the status ERROR, an updated message has the status FIXED, and a reinjected message has the status CLOSED.)

3. Build the Project and Test it Locally

  1. On the command line, run the following commands:

    mvn clean install
mvn spring-boot:run

  2. Demonstrate that your implemented logic is correct: there are records in the database, messages in the queues, etc.

4. Deploy to OpenShift

4.1. PostgreSQL database

When we deploy the project to OCP, we will use the PostgreSQL database and schema we set up earlier, instead of using the embedded H2 database. You can find the configuration details in the application.properties and deployment.yml files.

4.2. Deploying to OpenShift

Once the solution is tested locally, you can deploy to your project in OpenShift.

  1. In a terminal window, navigate to project root folder and run:

    mvn fabric8:deploy -Popenshift

  2. Check that the pod rest-publish-and-fix-errors-lab starts up correctly.

  3. Note the URL of the route.

4.3. Test the Solution on OpenShift

  1. Send a curl request to the REST endpoint and test that the messages flow correctly to the queues and topics.

  2. Check the PostgreSQL database console for the error messages in the T_ERROR table.

  3. Use the scripts below to fix the records.

  4. Test that the records are now processed correctly to the accountQueue.

5. Useful SQL Scripts

DELETE FROM USECASE.T_ERROR;

SELECT * FROM USECASE.T_ERROR;

UPDATE USECASE.T_ERROR SET MESSAGE='Error,EU,true,Fred,Quicksand,202 Barney Blvd.,Rock City,MI,19728,313-555-1234', STATUS='FIXED' WHERE ID=8;
